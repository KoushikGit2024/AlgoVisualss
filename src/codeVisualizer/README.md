# The AlgoVisuals Code Visualizer UI

> **Navigation:** [Root README](../../README.md) | [Interpreter Architecture](../interpreter/README.md) | [Feature Specs](../../FEATURES.md)

Welcome to the **Code Visualizer** module. This is the React-based frontend layer that acts as the "Player" for the C++ Execution Engine. It is entirely responsible for managing the user interface, interpreting the raw memory snapshots generated by the engine, and dynamically rendering interactive data structures to the screen.

## 1. Core Philosophy

**Separation of Concerns:**
The UI layer is strictly separated from the parsing and execution logic. The components in this folder do not parse ASTs or evaluate C++ math. Instead, they act as a **dumb player**, consuming an array of `RuntimeSnapshot` objects.

By keeping the visualizer completely agnostic to the actual C++ code, we achieve two things:
1. The visualizer can easily step forwards and backwards through time instantly.
2. We can theoretically drop in a Python or Java execution engine later, and as long as it emits standard `RuntimeSnapshot` objects, the UI will work perfectly without a single change.

---

## 2. Architecture Data Flow

When a user clicks **"Simulate"**, the data flows through the visualizer layer as follows:

1. **Input:** The `CodeEditor.tsx` holds the raw C++ string.
2. **Web Worker Offloading:** `VisualGround.tsx` spins up a dedicated Web Worker (`EngineWorker`) so the heavy AST parsing and IR execution does not block the React main thread.
3. **Snapshot Ingestion:** The worker returns an array of `RuntimeSnapshot` objects.
4. **Heuristic Detection:** As the user scrubs through time, the current snapshot's variables are passed to `detectVisualizer.ts`. This engine uses Regex/Prefix matching and Shape Validation to decide which data structures should be mounted.
5. **Rendering:** `VisualGround.tsx` dynamically mounts the necessary SVG/DOM components (like `Graph.tsx` or `Tree.tsx`) inside floating `DraggableWindow` containers.

## 4. The Master Orchestrator (`VisualGround.tsx`)

File: `sideComponents/VisualGround.tsx`

This is the most complex UI component in the application. It handles several distinct responsibilities:

### A. Background Thread Management
Because C++ algorithms (like deep recursion) can easily lock up a browser tab, `VisualGround` runs the `ExecutionEngine` inside a Web Worker. If the user spams the "Simulate" button, it forcefully terminates the old worker thread and spins up a new one to guarantee UI responsiveness.

### B. The Playback Timeline
It manages the `currentStep`, `isPlaying`, and `speed` states. A `setInterval` loop drives the playback forward, stepping through the `snapshots[]` array. The user can also manually scrub the timeline to jump to any point in time.

### C. Live Memory Tracking
It renders the right-hand panel consisting of:
- **Call Stack:** Shows the active C++ function frames.
- **Variables Watcher:** Shows every variable in the current local scope. It highlights variables in **Green** if they were just written to, and **Blue** if they were just read from, based on the current snapshot event.
- **Terminal Output:** Reconstructs the `std::cout` stream chronologically up to the `currentStep`.

## 5. The Heuristic Engine (`detectVisualizer.ts`)

File: `sideComponents/detectVisualizer.ts`

AlgoVisuals does not force the user to learn a custom Visualizer API. Instead, it magically guesses what data structures to render by looking at standard competitive programming habits.

The detection runs in a **Two-Phase Pass** on every frame:

### Phase 1: Prefix Matching
The engine iterates over every active variable name and checks for known prefixes:
- `adj_`, `graph_` → Candidate for Graph
- `dp`, `grid`, `mat` → Candidate for 2D Array
- `st_`, `stack` → Candidate for Stack
- `tree_`, `root_` → Candidate for Tree

### Phase 2: Shape Validation
A prefix match is not enough. The engine deep-unwraps the JavaScript value and validates its physical shape:
- If `adj_list` is just an integer, it is ignored. It must be a 2D array or edge list.
- If `tree_nodes` is detected, it must be an array of objects containing `{id, value, left, right}`. (We also auto-normalize positional tuples like `[id, val, left, right]` into objects).

### Pointer Collection
The detector also scans for auxiliary variables like `i`, `j`, `curr`, `left`, `right`. If it finds them, and their values correspond to an index or node ID in a detected structure, it bundles them into the `pointers` array. The data structure component will then render these as floating badges pointing at the correct nodes.

## 6. The Visual Primitives (`dataStructures/`)

The files in `dataStructures/` are strictly presentation-layer components.
- They **do not** hold algorithmic state.
- They **do not** know what step the player is on.
- They only accept a strict prop contract (e.g., `value: number[], pointers: {name, index}[]`).

Animations are handled via `framer-motion` `<motion.div>` tags using `layout` props. When an array element swaps index `2` with index `5`, `framer-motion` detects the key change and physically animates the blocks sliding past each other.

## 7. OS-Style Window Manager (`DraggableWindow.tsx`)

Because a user might have a Graph, a Queue, and a Visited Array all rendering at the same time, the canvas would quickly become cluttered.

To solve this, `VisualGround` mounts each detected structure inside a `DraggableWindow`. 
- **Z-Indexing:** Clicking a window brings it to the front.
- **Minimization:** Windows can be minimized into a bottom taskbar to hide visual noise.
- **Auto-Tiling:** When the visualizer first boots up, it detects how many structures exist and attempts to auto-tile them cleanly across the canvas to prevent them from spawning on top of each other.
